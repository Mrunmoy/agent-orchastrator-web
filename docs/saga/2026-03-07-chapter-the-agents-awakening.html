The Agents' Awakening

Saga Chapter · 2026-03-07 · UI-009

Prologue: The Dormant Roster

The cockpit was nearly complete. The conversations pane hummed on the left, the debate theatre stretched across the centre, and the intelligence pane kept its watchful vigil on the right. Yet something was wrong in the Agents section — a quiet, almost invisible wrongness.

Anyone who wandered to the bottom of the sidebar would find the roster: a small sign reading No agents configured, and a single button — Add Agent. They would press it. A panel would unfold at the foot of the page like a scroll unrolling on a stone floor. Name. Provider. Model. Role. Then they would stare at the Model field.

It was blank. Invitingly, ominously blank. What does one type there? A secret incantation? The name of a god? There was no clue, no placeholder, no hint.

So they would leave it empty. They would fill their agent's name — perhaps Codex Worker, perhaps Claude Reviewer — and they would click Save Agent. Nothing would happen. No agent would appear. No error would speak. The form would simply wait, mute and unmoved, like a locked gate with no keyhole in sight.

Act I: The Investigation

The ranger traced the failure to its lair in AppShell.tsx. The openCreateAgent function had been written with honest intentions — it opened the editor and set the initial model to an empty string. But the saveAgent function, equally well-intentioned, checked whether model was truthy before proceeding. An empty string is not truthy. Every save attempt was therefore rejected at the gate.

The rejection was logged — but only as a red banner at the very top of the application shell, far above the form that had caused it. The form sat at the bottom. The error lived at the top. Mortal eyes scrolled nowhere, and so the failure was invisible.

Deeper in the dungeon, a second disappointment lurked. The backend had supported personality_key since the first schema migration. The personalities.json config file listed three archetypes — Software Developer, Code Reviewer, Friendly Brainstormer — each with traits and instructions baked in. Yet the form had never offered a personality field. Every agent created through the UI was born without a soul.

Act II: The Forging

The forge was lit in AppShell.tsx. First, the ranger hammered out the PROVIDER_MODELS catalogue — a lookup table mapping each provider to its curated list of known models. Claude's roster of four flagship models. Codex's quartet of reasoning engines. Ollama's open-source menagerie. Gemini's flash and pro variants.

The Model field was recast from a free-text input into a dropdown <select>, its options drawn directly from the catalogue. When the form opened for a new agent, the first model of the default provider was pre-selected — so the form was ready to save before the user had typed a single character.

A subtle but important enchantment was woven into the Provider dropdown: when its value changed, the Model dropdown would reset in the same state update — atomically, invisibly — to the first model of the newly chosen provider. No stale Codex model would survive a switch to Ollama. Each agent would carry consistent lineage from the moment of its creation.

Then came the personality field. The PERSONALITY_OPTIONS array was conjured — a simple list of key-label pairs, with an empty-string sentinel at the front labelled — none —. A new <select> was added to the editor grid. When the user picked a personality, its key was conditionally spread into the API payload. When they left it on "none", no key was sent — and the backend column stayed null, as the schema intended.

Act III: The Agents Speak

With the editor fixed, attention turned to the roster itself — the cards that would one day populate the sidebar once agents had been created. Each card bore a small coloured dot: green for idle, amber for running, red for blocked, grey for offline. These dots were previously mute ornaments. Now they gained a voice.

A title attribute was added to each dot, so hovering would reveal a tooltip: Idle, Running, Blocked, Offline. An aria-label was added alongside for those who navigate without a cursor. The dot was no longer decorative; it was information.

And when an agent carried a personality, the card would now display a soft purple badge beside the provider and role labels. Software Developer. Code Reviewer. Friendly Brainstormer. The agent's character, visible at a glance.

Epilogue: The Roster Stirs

Three new tests joined the suite — one verifying that the model dropdown resets when the provider changes, one confirming that all four personality options are present and selectable, and the original "creates an agent" test updated to match the new dropdown reality. The full suite of 146 tests turned green. The linter found nothing to fault.

The form was no longer a locked gate. The roster was no longer empty by fate. Agents could now be born with names, providers, models, roles, and personalities — ready to take their place in the debate theatre when the next conversation called them forward.

The cockpit was a little more complete. The quest continued.
