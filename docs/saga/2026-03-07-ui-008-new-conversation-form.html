The Invisible Door

Saga Chapter · UI-008 · 2026-03-07
Every dungeon has a door that looks like a wall. The party had been staring at the "+ New" button for days, clicking it with growing suspicion, watching the Conversations panel remain stubbornly empty. Was it broken? Was it them? Turns out — the door was real. It just opened into a void.

The Haunted Button

The user filed their report: "Clicking on the +New button seems like doing nothing." A screenshot arrived. There it was — the purple "+ New" button, gleaming in the sidebar, doing absolutely nothing a user could see. Behind the scenes the button was creating conversations, auto-naming them Conversation 1 and inheriting whatever working directory happened to be in localStorage. But without a backend running, it failed silently. And even when it succeeded, there was no way to name the conversation or point it at a project directory.

The button had always technically worked. It just hadn't ever been useful.

The Fix: A Door with a Knocker

The party's ranger studied the existing AgentEditorState pattern — already used for the Add/Edit Agent workflow — and recognised the shape of the solution. The fix didn't need a new modal system or a separate page. It needed a form, the same kind the party had already built for agents: an inline section that appears below the main grid, pre-filled with sensible defaults, closeable with Cancel if the user changes their mind.

Three new pieces of state appeared in AppShell:

  • A ConversationCreatorState type — just a title and a working directory.
  • openConversationCreator() — triggered by "+ New"; opens the form with pre-filled defaults.
  • submitConversationCreator() — validates both fields are non-empty before firing the API.

And a helper appeared, quietly solving a subtle duplication that the ranger had noticed: three different places in the code all generated "Conversation N" independently. The generateDefaultTitle(count) function united them.

The Per-Conversation Working Directory

The user had also noticed something important: the global Working Directory field in the top bar was useless in practice. Each conversation needs its own project directory. Agents must be spawned in the right folder. Using a global setting silently applied the same directory to every conversation — which only works if you're always working on one project.

The new form surfaced the working directory at conversation creation time, letting the user see and change it before any agents are started. The global directory in TopBar now acts as a seed for the default value, not a runtime setting.

The Tests Come First

True to the campaign rules, the tests were written before the implementation. Three new assertions captured the intended behaviour:

  • Clicking "+ New" shows a form — no API call yet.
  • Clicking "Cancel" closes the form — still no API call.
  • Filling the form and clicking "Create" calls the API with the correct title and directory, then dismisses the form.

The gatekeeper (CI) accepted the offering. All 146 tests passed.

Loot Gained

  • ConversationCreatorState — new inline form type in AppShell.
  • generateDefaultTitle(count) — helper eliminating title duplication.
  • .conversation-creator CSS block — consistent with the agent editor panel.
  • 3 new tests, 2 updated tests — the suite grows to 146.
The door now has a knocker. When you click it, a form appears — modest, focused, ready to receive a name and a path. The party can finally open new conversations with intent.
