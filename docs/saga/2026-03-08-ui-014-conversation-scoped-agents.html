The Turn Order

Saga Chapter · 2026-03-08 · UI-014

The orchestrator had grown strong. Conversations could be created, agents spawned, messages sent. But something was wrong in the roster hall.

Two agents stood side by side, both wearing the badge numbered "1". The calude-code agent and the codex-code agent had been summoned into the same conversation, yet neither knew who should speak first. The turn order was broken — or rather, it had never existed.

The Discovery

The agents had been global citizens. They lived in a flat table, unaware of which conversation had summoned them. A conversation_agent join table had been carved into the schema from the beginning — complete with a turn_order column — but no one had ever written to it. It was a door built into a wall with nothing on the other side.

The Forge

Tests were written first, as was the way. Sixteen tests described the world as it should be: agents listed by turn order, auto-incrementing positions, scoped per conversation, removal without destruction, bulk reordering by decree.

The schema was reforged. Migration v3 dropped the empty join table and rebuilt it with foreign key constraints binding agents to their conversations, a uniqueness seal preventing duplicates, and cascade rules ensuring no orphans would linger when a conversation or agent was destroyed.

Four endpoints emerged: one to list, one to create-and-link, one to sever the bond, and one to reorder the ranks. The POST /remove was corrected to DELETE during review — a reminder that REST has its own grammar.

The Order Restored

Now when agents are summoned to a conversation, each receives a turn number. The first agent is 1, the second is 2, the third is 3. The round-robin knows who speaks next. The badges in the roster hall finally show different numbers, and the cycle turns as it should.

552 tests passed. The forge holds.
