Gotcha: model field was empty by default → silent save failure

Symptom: Clicking "Save Agent" appeared to do nothing. No agent was created.

Cause: openCreateAgent initialised model: "". The guard if (!payload.model) in saveAgent fired immediately and set an error string — but the error banner is rendered at the top of AppShell, far above the editor panel at the bottom. Users never saw it.

Fix: Model is now a <select> pre-populated with the first entry of PROVIDER_MODELS[provider]. The model value is always non-empty when the form opens, so save succeeds as soon as the user fills in a name.

Gotcha: provider and model can get out of sync on edit

Symptom: If a user changes the provider dropdown, the model dropdown still shows models from the previous provider.

Fix: Provider onChange atomically updates both provider and model in a single setAgentEditor call: model: PROVIDER_MODELS[newProvider][0].

Edge case: When editing an existing agent whose model string is not in the PROVIDER_MODELS list (e.g. a custom Ollama model), it is injected as an extra <option> so the select renders correctly without losing the value.

Gotcha: personality_key "" vs null vs undefined

The backend personality_key column is nullable. Sending an empty string would store "" which is technically valid but semantically wrong. The saveAgent function conditionally spreads: ...(agentEditor.personality_key ? { personality_key: agentEditor.personality_key } : {}). When "— none —" is selected (empty string), no key is sent and the backend column remains NULL.
