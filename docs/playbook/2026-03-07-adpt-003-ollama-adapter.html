Ollama Memo Adapter Notes

Playbook Entry · ADPT-003 · 2026-03-07

Key Difference: Module Functions vs. Class

The Ollama adapter uses module-level run() and check_available() functions with an OllamaAdapterConfig dataclass, rather than inheriting from BaseAdapter. This reflects its stateless memo role — no session tracking, no instance state.

Key Difference: Prompt via stdin

Claude and Codex pass the prompt as a CLI argument. Ollama receives the prompt via stdin pipe (proc.communicate(input=prompt.encode())). This avoids shell escaping issues with large debate transcripts containing special characters.

Key Difference: Plain Text Output

Claude and Codex return JSON that must be parsed. Ollama returns plain text directly from stdout. No JSON parsing is needed.

Host Configuration

The OLLAMA_HOST environment variable is passed to the subprocess, allowing connection to remote Ollama instances without modifying global environment.

Key Files

  • backend/src/agent_orchestrator/adapters/ollama_adapter.py — implementation
  • backend/tests/test_ollama_adapter.py — 15 tests covering config, run, timeout, errors, stdin piping, availability

Running Tests

  • cd backend && python -m pytest tests/test_ollama_adapter.py -q
  • All tests mock asyncio.create_subprocess_exec — no real Ollama install required.

Gotchas

  • Use TimeoutError (builtin), not asyncio.TimeoutError — ruff UP041 flags the alias.
  • The check_available() function catches both FileNotFoundError and OSError to handle cases where the binary is missing or the system call fails for other reasons.
  • The run() function also catches FileNotFoundError from create_subprocess_exec itself (before communicate), returning an error result rather than raising.
