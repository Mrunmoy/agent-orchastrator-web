Conversation-Scoped Agents

Playbook Entry · 2026-03-08 · UI-014

Symptom

Both agents in a conversation show badge "1". Creating multiple agents shows duplicate turn numbers.

Root Cause

Agents were stored globally with a sort_order column defaulting to 0. The conversation_agent join table existed in the schema but was never wired up.

Fix

  • Schema migration v3: recreate conversation_agent with FK constraints, UNIQUE(conversation_id, agent_id), and DEFAULT on turn_order
  • New endpoints: GET /conversations/{id}/agents, DELETE /conversations/{id}/agents/{aid}, PATCH /conversations/{id}/agents/reorder
  • Modified POST /agents to accept optional conversation_id with auto-incrementing turn_order
  • Cascade delete: deleting an agent cleans up all conversation_agent rows

Watch Out For

  • Existing databases need migration v3 to get the updated conversation_agent table
  • The turn_order is 1-based (first agent = 1, not 0)
  • Deleting a conversation cascades to conversation_agent rows via FK
  • The global agent.sort_order still exists but is separate from per-conversation turn_order
