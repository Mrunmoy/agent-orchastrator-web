Schema Migration Behaviour

The migration in db.py::_apply_migrations() checks PRAGMA table_info(agent) before issuing ALTER TABLE:

existing = {row[1] for row in conn.execute("PRAGMA table_info(agent)").fetchall()}
if "sort_order" not in existing:
    conn.execute("ALTER TABLE agent ADD COLUMN sort_order INTEGER NOT NULL DEFAULT 0")
    conn.commit()
This migration is safe to run multiple times. New databases created from schema.sql already have the column; the migration skips for them.

Troubleshooting: Render-Phase State Sync

AgentRoster uses a render-phase update pattern (prevAgentsRef comparison) rather than useEffect to sync prop changes to local state. This prevents a missed re-render cycle when a new agent is added via the editor. If you see stale display after agent creation:

  1. Verify prevAgentsRef.current !== agents triggers on the new render.
  2. Ensure setAgents in AppShell produces a new array reference (it does via spread).

Adding a Bulk Reorder Endpoint (Follow-Up)

The current implementation calls PATCH /agents/{id}/order once per agent on each drop. For rosters larger than ~10 agents, a bulk endpoint is preferred:

PATCH /agents/reorder
Body: {"order": ["id-1", "id-2", "id-3"]}

The backend would UPDATE all agents' sort_orders in a single transaction. The frontend would call this once per drop event.
