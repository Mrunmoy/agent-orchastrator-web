Agent Editor UX Improvements (UI-009)

Design Codex Entry · 2026-03-07

Problem

The "Add Agent" form (rendered as a panel below BottomControls) had three critical UX failures:

  1. Model field was a blank free-text input — users had no guidance on what to type. There are dozens of valid model identifiers per provider and typos produce silent failures at inference time.
  2. No Personality field — the backend API and personalities.json already supported personality_key, but the form never surfaced it.
  3. Save Agent did nothing — the model field defaulted to an empty string, causing the required-field guard (if (!payload.model)) to fire on every submit. The resulting error was rendered far above the editor panel and invisible without scrolling, so the form appeared silently broken.

State Shape

The AgentEditorState type in AppShell.tsx was extended with a personality_key: string field (empty string for "none"). Both openCreateAgent and openEditAgent populate the field; saveAgent conditionally spreads it into the API payload.

Decisions

  • Model list is a static frontend constant rather than a live API call to keep the form snappy and avoid a loading state. The list can be extended easily in one place (PROVIDER_MODELS).
  • Personality is optional — sending an empty personality_key would create confusion at the backend. The conditional spread (...(key ? { personality_key: key } : {})) avoids it.
  • Status icon tooltip is implemented via HTML title attribute for zero-cost hover UX; aria-label adds screen-reader coverage.
  • Personality badge uses a distinct purple colour to visually separate it from the provider badge (also purple-family) by shade.

Follow-Up

  • Model list could be fetched from a backend endpoint (e.g. /agents/models?provider=ollama) to stay current with locally installed Ollama models.
  • The agent editor is currently a full-width bottom panel; a modal dialog would provide better focus management and keyboard trap behaviour.
  • Agent status (idle / running) should update in real time via the events stream endpoint (API-005) once the live-update feature is wired into the UI.
