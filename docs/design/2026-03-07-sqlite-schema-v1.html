SQLite Schema v1

Design Codex Entry · 2026-03-07 · DATA-001

Context

The orchestrator requires persistent state for conversations, agents, tasks, scheduler runs, and system resource snapshots. SQLite in WAL mode was chosen for local-first performance with concurrent read access.

Schema Entities

  • agent — registered CLI agent instances (provider, model, role, capabilities).
  • conversation — long-lived orchestration workspaces with state machine phase tracking and soft delete.
  • conversation_agent — join table binding agents to conversations with turn order and permission profiles.
  • task — work items within conversations supporting dependency chains and ownership.
  • artifact — agreement maps, conflict maps, neutral memos, and checkpoints for smart resume.
  • message_event — ordered event stream with unique event_id and autoincrement primary key.
  • scheduler_run — batch execution lifecycle records.
  • resource_snapshot — CPU/RAM/GPU telemetry for capacity-aware throttling.

Decisions

  • All IDs are TEXT (UUID-style) except autoincrement counters on message_event and resource_snapshot.
  • JSON fields use _json suffix convention for clarity (capabilities_json, spec_json, etc.).
  • Timestamps stored as ISO 8601 text for portability and human readability.
  • Six indexes match the recommended query patterns from the domain model spec.
  • Schema uses CREATE TABLE IF NOT EXISTS / CREATE INDEX IF NOT EXISTS for idempotent application.

Rationale

  • WAL mode enables concurrent readers without blocking the single writer path.
  • PRAGMA synchronous = NORMAL balances durability with write performance for local-only operation.
  • Keeping JSON in text columns avoids schema rigidity for evolving capability/metadata payloads.
