Agent Drag-to-Reorder & Composer Cleanup (UI-010)

Design Codex Entry · 2026-03-07

Problem

Two related UX issues existed in the orchestrator interface:

  1. The "Target First" agent-selector dropdown in the Composer — a static, hardcoded dropdown that bore no relationship to the actual configured agents and served no useful purpose in the current phase-gate orchestration model where agents reply in round-robin order.
  2. No way to reorder agents — agents appeared in alphabetical order only. The orchestrator processes agents in turn_order, so users needed a way to visually control and persist agent ordering in the left pane.

Frontend: Drag-and-Drop Design

Uses the browser's native HTML5 drag-and-drop API — no third-party library dependency:

  • dragstart: records the dragged agent ID in a useRef (avoids stale closure issues).
  • dragover: calls preventDefault() to allow drop.
  • drop: splices array locally (optimistic update), reassigns sort_order by index, fires reorderAgent() for each agent asynchronously.
  • A visual drag handle ⠿ is shown at the left of each card wrapper.
  • reorderAgent is an injectable prop for testability (defaults to the imported client function).

Order Badge Design

  • Positioned absolute top-left of .agent-card (which gains position: relative).
  • 18×18 px filled circle, background: #d6b164 (brand gold), white bold text, font-size 10px.
  • Shows agent.sort_order + 1 so display is 1-indexed (first agent shows "1").
  • data-testid="order-badge" for reliable test selection.

Composer Changes

The ComposerProps interface loses agents as a meaningful input and onSend simplifies to (message: string) => void. The agents prop is retained as an empty optional array for backward compatibility but renders nothing. ChatPane's inline composer also drops its hardcoded <select>.

Migration Safety

The schema migration uses PRAGMA table_info(agent) to check whether sort_order already exists before issuing ALTER TABLE. This makes the migration idempotent — safe to run on both new and existing databases.

Decisions

  • Native DnD preferred over libraries (dnd-kit, react-beautiful-dnd) to keep the bundle lean and avoid React 19 compatibility concerns.
  • Optimistic local reorder gives instant visual feedback; API failures are logged to console (no rollback in this slice — a follow-up can add error toast + rollback).
  • All agents' sort_orders are updated on drop (N calls) rather than attempting a complex swap algorithm. With a typical roster of 2–8 agents this is not a concern.
  • 1-indexed display ("1", "2", …) is more natural for users; 0-indexed storage is conventional for arrays.

Follow-Up

  • Rollback on API failure: display an error toast and restore previous order if reorderAgent rejects.
  • Sync sort_order with conversation_agent.turn_order so orchestration respects the user-defined sequence.
  • Consider a bulk PATCH /agents/reorder endpoint to update all sort orders in one round-trip.
