Migration/Bootstrap Loader

Design Codex Entry · 2026-03-07 · DATA-002

Context

The storage module had a schema.sql DDL file but no programmatic way to initialize a database, manage connections, or track schema versions. Application code had no entry point for bootstrapping.

Decisions

  • DatabaseManager class holds a single sqlite3.Connection internally.
  • initialize() reads and executes schema.sql, sets WAL mode on file-backed DBs, tracks version via PRAGMA user_version.
  • initialize() is idempotent — safe to call multiple times without data loss.
  • connection() context manager yields the internal connection with foreign keys enabled.
  • Supports both str and pathlib.Path for db_path. Also implements __enter__/__exit__ for use as a context manager.

Rationale

  • PRAGMA user_version is simpler than a schema_version table — built into SQLite.
  • Single persistent connection matches SQLite's single-writer design.
  • Alembic or similar migration tools are overkill for a single-schema local-first app.

Gotcha

executescript() resets PRAGMA foreign_keys to OFF. The implementation re-enables foreign keys after every executescript() call.

Test Strategy

14 tests in test_db_manager.py: initialization, idempotency, data preservation, connection management, WAL mode, foreign keys, context manager protocol, path object acceptance, close behavior, schema version edge cases.

Follow-Up

  • Connection pooling not needed now (single-writer model).
  • Multi-file migration chains deferred until schema v2.
