FastAPI App + Health Endpoint

Design Codex Entry · 2026-03-07 · API-001

Context

The backend package structure existed but contained no application instance or routes. Downstream tasks (orchestration API, WebSocket layer, frontend integration) needed a running HTTP server to build on.

Decisions

  • App factory pattern: create_app() in api/__init__.py builds FastAPI instance and includes routers. Module-level app = create_app() enables uvicorn import.
  • Two endpoints: GET /health (liveness probe) and GET /state (app metadata stub).
  • All responses follow the {"ok": true/false, ...} envelope convention. Shared helpers ok_response() and error_response() extracted to api/responses.py.
  • Synchronous endpoints — health/state are trivial, no async needed.

Rationale

  • App factory allows test isolation and future configuration injection.
  • Shared envelope helpers prevent duplication across future route modules.
  • Keeping app in api/__init__.py is conventional for small FastAPI projects.

Test Strategy

9 tests using fastapi.testclient.TestClient covering response status codes, JSON structure, and envelope fields for both endpoints.

Nix Change

Added python312Packages.httpx to flake.nix — required by TestClient.

Follow-Up
