Domain Models

Design Codex Entry · 2026-03-07 · ORCH-001

Context

The orchestration core needs canonical Python data structures that mirror the domain model spec (docs/specs/12-domain-model.md). These models define agents, conversations, tasks, artifacts, message events, scheduler runs, and resource snapshots. They are the shared vocabulary between the storage layer, the scheduler, and the API.

Decisions

  • All domain entities are Python @dataclass classes with full type hints.
  • Nine enums capture the finite state values: Provider, AgentRole, AgentStatus, ConversationState, Phase, GateStatus, TaskStatus, RunStatus, ArtifactType.
  • Enum values use lowercase strings matching the SQL column values (e.g., Provider.CODEX = "codex").
  • Optional fields use Optional[T] and default to None.
  • Fields with SQL defaults (like priority, active, enabled, JSON columns) carry matching Python defaults.
  • Timestamp fields are typed as str to match SQLite TEXT storage.
  • Integer primary keys (MessageEvent.id, ResourceSnapshot.id) are Optional[int] since they are auto-incremented.

Rationale

  • Dataclasses provide a clean, lightweight representation with no ORM coupling.
  • String-valued enums allow direct round-tripping with SQLite without conversion.
  • Defaults mirror SQL defaults so Python-constructed instances are DB-ready.
  • No frozen dataclasses here — domain models need to be mutable for state transitions and updates.

Entities

  • Agent — 11 fields: identity, provider/model, role/status, session, capabilities, timestamps.
  • Conversation — 13 fields: title, project path, state machine (state/phase/gate), priority, summary, timestamps, soft delete.
  • ConversationAgent — 8 fields: join table linking agents to conversations with turn order and permissions.
  • Task — 13 fields: conversation-scoped work items with status, ownership, dependencies, evidence.
  • Artifact — 6 fields: typed structured outputs (agreement maps, checkpoints, etc.).
  • MessageEvent — 9 fields: chat stream and event timeline entries.
  • SchedulerRun — 8 fields: batch execution lifecycle tracking.
  • ResourceSnapshot — 7 fields: machine resource telemetry for auto-throttle.

Test Strategy

TDD with tests covering: enum member completeness, dataclass instantiation with valid data, optional field defaults, enum field acceptance, and type consistency. All tests written before implementation.
