Capacity Telemetry Snapshot

Design Codex Entry · 2026-03-07 · OPS-003

Context

The scheduler needs real-time machine resource data to auto-throttle agent runs. Without capacity awareness, the system could overload the host by launching too many concurrent agents.

Decisions

  • CapacitySnapshot frozen dataclass mirrors resource_snapshot table fields: CPU load, RAM used/total, GPU stub, agent capacity, timestamp.
  • capture_snapshot() reads 1-min CPU load average (as percent of total cores) and RAM usage via psutil.
  • check_capacity() evaluates three ordered gates: CPU ≥ 85%, free RAM ≤ 2048 MB, active runs ≥ max. First failing gate produces a denial with a descriptive reason.
  • All thresholds are configurable via function parameters with sensible defaults exposed as module constants.

Rationale

  • Synchronous psutil reads are fast enough — no async overhead needed.
  • Frozen dataclasses ensure snapshots are immutable once captured.
  • Ordered gate evaluation gives deterministic denial reasons for debugging.

Test Strategy

22 tests with fully mocked psutil: dataclass field access, immutability, each gate at below/at/above threshold, custom thresholds, gate priority ordering, all-pass happy path.

Follow-Up

  • GPU telemetry stubbed as "[]" — enumerate GPUs when needed.
  • Scheduler integration (ORCH-006) will call check_capacity() before launching runs.
